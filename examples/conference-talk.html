Part 2: The Architecture

Part 2: The ArchitectureDistributed Systems Conference 2025

Single engine, multiple models

One storage layer, one query planner, one transaction manager — many access patterns.

  • Storage Engine: LSM-tree with pluggable column families for each model
  • Query Planner: Model-aware cost-based optimizer — picks the best index per query pattern
  • Transaction Manager: Serializable snapshot isolation across models, same MVCC layer
  • Index Layer: B-tree for relational, inverted for text, HNSW for vectors, adjacency list for graph
  • Wire Protocols: PostgreSQL wire protocol, Redis RESP, HTTP/GraphQL

Part 3: Lessons Learned

Part 3: Lessons LearnedDistributed Systems Conference 2025

What broke (and how we fixed it)

  1. LSM compaction storms at 2TB per node → tiered compaction + rate limiting
  2. HNSW graph corruption on crash → write-ahead log for index mutations
  3. Cross-model deadlocks → global wait-for graph with timeouts
  4. Postgres wire protocol edge cases → fuzzed with 10K+ random protocol messages
  5. Memory pressure under mixed workloads → adaptive cache sizing per model
What broke (and how we fixed it)Distributed Systems Conference 2025
Do not build a multi-model database. Pick one model. Make it fast. Then add the next one. If you try to build everything at once, you will build nothing at all.
